# Git workflow

## Commits

Conventional Commits style, matching existing history: `feat:`, `fix:`,
`docs:`, short imperative subject, no body unless truly needed, no
co-author trailers. For a `fix:` commit, the subject must describe the
actual bug ("fix: message snippet truncation breaking utf-8"), not a vague
placeholder like "fix bugs" or "misc fixes".

Commits should carry a DCO sign-off (`git commit -s`). An agent committing
on the user's behalf should ask whether to include `-s` rather than adding
it (or leaving it out) silently.

## Branches

- One branch per feature/fix: `type/short-slug` (e.g.
  `feat/vim-mode-tables`, `fix/oauth-keyring-limit`). The slug should be
  descriptive of the actual bug or feature, not generic. Keep unrelated
  changes out of a branch.
- `dev` is the integration branch: feature/fix branches merge into `dev`,
  and `dev` merges into `main` for releases. Don't merge straight into
  `main` for day-to-day work.
- Don't force-push shared branches (`main`, `dev`) or rewrite published
  history without explicit sign-off.

## Release changelog

Release notes are auto-generated by GitHub from merged PRs, configured in
`.github/release.yml` (categories: Features / Bug Fixes / Other Changes,
with `documentation` and `no-changelog` labels excluded entirely). PR labels
matching the conventional-commit type are applied automatically by
`.github/workflows/conventional-prs.yml`, so PR titles must follow the
`type: subject` convention for both title validation and changelog
categorization to work.
